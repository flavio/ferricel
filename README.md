# Ferricel

A CEL (Common Expression Language) to WebAssembly compiler written in Rust.

## Overview

Ferricel compiles CEL expressions into standalone WebAssembly modules that can be executed in any Wasm runtime. Each compiled Wasm module includes an embedded runtime library, making them fully self-contained and portable.

## Features

- **CEL to Wasm compilation**: Compile CEL expressions into efficient WebAssembly bytecode
- **Standalone modules**: Each Wasm file includes its own runtime - no external dependencies
- **Protocol Buffer support**: Full support for protobuf messages with proper wrapper type semantics
- **Type safety**: Optional type checking at compile time when proto schemas are provided
- **Fast evaluation**: Linear time complexity with respect to expression and input size

## Quick Start

### Basic Usage

```bash
# Build a simple CEL expression
ferricel build --expression "1 + 2 * 3" --output expr.wasm

# Run the compiled expression
ferricel run expr.wasm
# Output: 7
```

### Using Variables

```bash
# Expression with variables
ferricel build --expression "name + ' is ' + age + ' years old'" --output greeting.wasm

# Provide variables at runtime (coming soon)
ferricel run greeting.wasm --input '{"name": "Alice", "age": 30}'
```

## Protocol Buffer Support

Ferricel supports Protocol Buffer messages with full wrapper type semantics as defined by the CEL specification. This includes proper handling of:

- **Wrapper-to-primitive equality**: `BoolValue{value: true} == true` → `true`
- **Empty wrapper zero values**: `BoolValue{} == false` → `true`
- **Unset wrapper field access**: `TestAllTypes{}.single_bool_wrapper == null` → `true`
- **Wrapper types are not null**: `BoolValue{} != null` → `true`

### Generating Proto Descriptors

To enable proto support, you need to provide a FileDescriptorSet (binary proto descriptor) generated by `protoc`:

```bash
# Generate a proto descriptor from your .proto files
protoc --descriptor_set_out=types.pb \
       --include_imports \
       path/to/your/message.proto

# Use it when compiling CEL expressions
ferricel build \
    --proto-descriptor types.pb \
    --expression "MyMessage{}.wrapper_field == null" \
    --output expr.wasm
```

### Why FileDescriptorSet?

Ferricel uses `FileDescriptorSet` (the binary proto descriptor format) rather than parsing `.proto` source files directly. This approach:

- **More reliable**: Uses Google's canonical `protoc` compiler for parsing
- **Standard format**: The same format used internally by Protocol Buffer tooling
- **No parsing complexity**: Ferricel only needs to deserialize, not parse proto syntax
- **Guaranteed correctness**: No risk of parser incompatibility or edge cases

### Example: Working with Proto Messages

Given this proto file (`person.proto`):

```protobuf
syntax = "proto3";

package example;

import "google/protobuf/wrappers.proto";

message Person {
  string name = 1;
  int32 age = 2;
  google.protobuf.StringValue nickname = 3;  // Wrapper type
}
```

Generate the descriptor and use it:

```bash
# Step 1: Generate descriptor
protoc --descriptor_set_out=person.pb \
       --include_imports \
       person.proto

# Step 2: Compile CEL with proto support
ferricel build \
    --proto-descriptor person.pb \
    --expression "Person{name: 'Alice', age: 30}.nickname == null" \
    --output check_nickname.wasm

# Step 3: Run
ferricel run check_nickname.wasm
# Output: true (because nickname wrapper field is unset)
```

### Multiple Proto Files

If your CEL expressions use types from multiple proto files, you can provide multiple descriptors:

```bash
# Option 1: Merge during protoc compilation
protoc --descriptor_set_out=all_types.pb \
       --include_imports \
       types1.proto types2.proto types3.proto

ferricel build \
    --proto-descriptor all_types.pb \
    --expression "..." --output expr.wasm

# Option 2: Provide multiple descriptors to ferricel
protoc --descriptor_set_out=types1.pb --include_imports types1.proto
protoc --descriptor_set_out=types2.pb --include_imports types2.proto

ferricel build \
    --proto-descriptor types1.pb \
    --proto-descriptor types2.pb \
    --expression "..." --output expr.wasm
```

### When is Proto Schema Required?

You only need to provide proto descriptors if your CEL expressions:

1. Use Protocol Buffer message types
2. Access wrapper type fields (`google.protobuf.*Value` types)
3. Need proper wrapper semantics (equality, null handling)

Simple expressions work without schemas:

```bash
# No schema needed - simple types
ferricel build --expression "1 + 2 * 3" --output expr.wasm

# No schema needed - maps and lists
ferricel build --expression "{a: 1, b: 2}.a + [1,2,3].size()" --output expr.wasm
```

### Warning: Missing Schema

If you compile a CEL expression that looks like it uses protobuf types (contains package names with dots) but don't provide a schema, ferricel will emit a warning:

```bash
$ ferricel build --expression "cel.expr.TestAllTypes{}.field" --output expr.wasm
Warning: Struct 'cel.expr.TestAllTypes' looks like a protobuf message, but no schema provided.
         Wrapper type semantics will not be available.
         Use --proto-descriptor to provide schema.
```

The compilation will succeed, but wrapper type semantics won't work correctly. To fix this, provide the proto descriptor.

## CEL Specification Coverage

Ferricel targets full compliance with the [CEL specification](https://github.com/google/cel-spec) and the [cel-go extension libraries](https://pkg.go.dev/github.com/google/cel-go/ext). Conformance is validated against the official CEL conformance test suite (**1458/1460 passing, 99.9%**). The two remaining failures are parser-level bugs in the upstream `cel` crate (negative hex literals and a string escape sequence edge case).
